## `GET /generations`

### Query parameters

<ParamField query="page" default="1" type="integer">
  1-indexed page number. Must be a positive integer.
</ParamField>

<ParamField query="limit" default="250" type="integer">
  Page size, between **1 and 250**.
</ParamField>

<ParamField query="status" type="string">
  Optional filter. One of `pending`, `completed`, `failed`.
</ParamField>

### Example

```text
GET /generations?status=failed&limit=20&page=2
```

### Response notes

- **`per_page`** always reflects the `limit` actually used â€” use it for pagination math. Never assume 250.
- **`tool` and `model`** are point-in-time snapshots stored on the generation itself, **except** `model.name`, which is looked up live and is absent entirely if the model has no translations.
- **`result_urls`** is present only when `status: "completed"`.
- **`error_message`** is present only when `status: "failed"`.

### Reading `credits_logs`

The ledger rows tell you exactly what happened to the wallet:

| Rows on a generation | Meaning |
| --- | --- |
| `reserve` only | Still pending â€” credits are held. |
| `reserve` \+ `spend` | Completed â€” charge was kept. |
| `reserve` \+ `release` | Failed / cancelled â€” credits were refunded. |
